Origins & history

RSA (Rivest–Shamir–Adleman, 1977) is a trapdoor permutation over \(\mathbb{Z}_n^*\). OAEP (Optimal Asymmetric Encryption Padding) was proposed by Bellare–Rogaway (1994/95) to transform a trapdoor permutation into a probabilistic, semantically secure PKE in the Random Oracle Model (ROM). Shoup (2001) revisited the proof for general TDPs; subsequent results establish CCA2 security of RSA–OAEP under RSA-type assumptions in the ROM. PKCS #1 v2.2 (RFC 8017, 2016) specifies RSAES-OAEP precisely; NIST SP 800-56B and SP 800-57 provide parameter guidance.

Design overview

Naïve RSA is deterministic \(c=m^e\bmod n\), leaking equality and enabling malleability. OAEP wraps the message in a randomised encoding using two masks derived via MGF1 from a fresh seed and the data block. The resulting encoded message (EM) is then mapped to an integer and exponentiated. Correct decoding removes the masks and checks fixed headers, providing IND-CCA2 security in the ROM when combined with RSA.

Mathematical setting

RSA keys

Choose primes \(p,q\); set \(n=pq\), \(\lambda(n)=\mathrm{lcm}(p-1,q-1)\). Pick \(e\) with \(\gcd(e,\lambda(n))=1\) (commonly \(65537\)), and compute \(d\equiv e^{-1}\pmod{\lambda(n)}\). Encryption/decryption:

\( c \equiv m^{e}\ (\bmod\ n),\qquad m \equiv c^{d}\ (\bmod\ n). \)

OAEP components

Fix a hash \(H\) (output \(hLen\) bytes) and MGF1 (based on \(H\)). For label \(L\), set \(\mathtt{lHash}=H(L)\).

EME-OAEP encoding

Let \(k\) be the modulus length in bytes and \(mLen=|M|\). Valid messages satisfy

\( mLen \le k - 2\,hLen - 2. \)

  1. Form \(\mathtt{DB}=\mathtt{lHash}\ \| \ \mathtt{PS}\ \|\ \texttt{0x01}\ \| \ M\), with \(\mathtt{PS}\) all-zero padding.
  2. Sample random seed \(S\in\{0,1\}^{8\,hLen}\).
  3. \(\mathtt{maskedDB} = \mathtt{DB} \oplus \mathrm{MGF1}(S,\ k-hLen-1)\).
  4. \(\mathtt{maskedSeed} = S \oplus \mathrm{MGF1}(\mathtt{maskedDB},\ hLen)\).
  5. \(\mathtt{EM}=\texttt{0x00}\ \|\ \mathtt{maskedSeed}\ \|\ \mathtt{maskedDB}\).

Decryption reverses masking, checks \(\mathtt{lHash}\) and the delimiter 0x01, then parses \(M\).

Parameter sets

Recommended choices

  • Hash \(H\): SHA-256 (or SHA-384/512 for higher strength).
  • MGF: MGF1 using the same hash as \(H\).
  • Label \(L\): usually empty and fixed per key.
  • Public exponent: \(e=65537\).

Message size: with modulus \(k\) bytes and hash \(hLen\) bytes, \(|M|\le k-2hLen-2\). Example: RSA-2048 (\(k=256\)) and SHA-256 (\(hLen=32\)) → max plaintext 190 bytes. In practice, RSA-OAEP encrypts a random session key for hybrid encryption.

Algorithms

Key generation

Generate \(p,q\); compute \(n, \lambda(n), e, d\); store CRT values \(d_p, d_q, q^{-1}\bmod p\) for fast private ops.

Encryption (RSAES-OAEP)

  1. Build \(\mathtt{EM}\leftarrow\mathrm{EME\mbox{-}OAEP\mbox{-}ENCODE}(M,L,H,\mathrm{MGF1})\).
  2. Output \( C = \mathrm{I2OSP}\big(\mathrm{OS2IP}(\mathtt{EM})^e \bmod n,\ k\big) \).

Decryption

  1. Recover \(\hat{\mathtt{EM}}=\mathrm{I2OSP}(C^d\bmod n,\ k)\).
  2. Parse and verify via \(\mathrm{EME\mbox{-}OAEP\mbox{-}DECODE}\); on any check failure, return a uniform error (constant-time).

Security at a glance

  • Goal: IND-CCA2 in the ROM when combined with RSA.
  • Assumptions: RSA inversion hardness; \(H\) and MGF1 modeled as random oracles.
  • Proof notes: Padded encoding plus trapdoor permutation yields tight security for RSA-OAEP (with RSA-specific refinements in the proofs).
  • Side-channels: use RSA blinding, constant-time exponentiation/parsing, and CRT fault checks (re-encrypt and compare).
  • Quantum note: RSA is vulnerable to Shor’s algorithm; prefer hybrid/PQ transitions for long-term confidentiality.

Implementation notes

  • Fix \(H\), MGF1, and label \(L\) per key; don’t mix parameters with the same key.
  • Use vetted libraries; enable CRT with verification; zeroise sensitive buffers.
  • Uniform error handling on decode; never surface padding-oracle differences.
  • Use RSA-OAEP to wrap symmetric keys; use AEAD (AES-GCM/ChaCha20-Poly1305) for bulk data.

Pros & cons

Pros

  • Widely standardized/interoperable (RFC 8017, NIST guidance).
  • Provable CCA2 security in ROM with clear parameterisation.
  • Excellent fit for hybrid key establishment.

Cons

  • Large keys/ciphertexts and slower private ops vs modern KEMs.
  • Implementation pitfalls (timing/oracles/faults) if countermeasures are absent.
  • No post-quantum security.
